## Critical issues (would burn in production)

1. Confirmed: the main app recording flow no longer uses `DictationPipeline`. `ContentView.stopAndTranscribe()` now transcribes and directly calls `finishTranscription()` ([App/ContentView.swift:218](App/ContentView.swift:218), [App/ContentView.swift:220](App/ContentView.swift:220)), and `finishTranscription()` appends only raw text with no cleanup/follow-up handling ([App/ContentView.swift:233](App/ContentView.swift:233)). That silently kills main-app cleanup, chained follow-ups, post-processing cancellation, and the shared intent/main-app behavior contract.
2. Confirmed: background `forceStop()` can destroy an in-flight manual stop. `ContentView` starts an async stop/drain at [App/ContentView.swift:204](App/ContentView.swift:204), but `JotApp` may enqueue `forceStop()` on background at [App/JotApp.swift:75](App/JotApp.swift:75); `forceStop()` nils `capture` and sets `isRecording = false` ([App/Recording/RecordingService.swift:311](App/Recording/RecordingService.swift:311), [App/Recording/RecordingService.swift:317](App/Recording/RecordingService.swift:317), [App/Recording/RecordingService.swift:320](App/Recording/RecordingService.swift:320)). If that wins the actor queue, `stop()` throws `.notRunning` instead of saving the dictation.
3. Confirmed: Action Button / Live Activity durations still include post-transcription LLM latency. The call sites snapshot `startedAt` before `stopAndTranscribe()` ([App/Intents/RecordAndTranscribeIntent.swift:182](App/Intents/RecordAndTranscribeIntent.swift:182), [App/Intents/DictateIntent.swift:189](App/Intents/DictateIntent.swift:189), [Shared/StopDictationIntent.swift:114](Shared/StopDictationIntent.swift:114)), but `DictationPipeline` waits for follow-up resolution before computing duration ([App/Intents/DictationPipeline.swift:145](App/Intents/DictationPipeline.swift:145), [App/Intents/DictationPipeline.swift:162](App/Intents/DictationPipeline.swift:162)).
