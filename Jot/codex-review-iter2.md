## Critical issues remaining (would burn in production)
1. Confirmed: main-app cleanup can hang the only persistence path. `ContentView` awaits cleanup after transcription and before `TranscriptStore.append`/clipboard publish (`App/ContentView.swift:220-255`), while `CleanupService.clean` awaits `LanguageModelSession.respond` with no deadline (`App/Cleanup/CleanupService.swift:140-151`). During that wait the record button is disabled as `.transcribing` (`App/ContentView.swift:419-434`). If Apple Foundation Models reports available but stalls, the user gets no saved raw transcript and no copy. This needs a timeout/fallback, or raw append before best-effort cleanup.
2. Confirmed/likely: purge/reload still has a real filesystem race. `purgeAndReload` cancels the old prepare task, waits only 500 ms, then deletes the shared model directory and immediately starts `warmUp()` (`App/Transcription/TranscriptionService.swift:258-285`). The old task may still be inside `AsrModels.download(to: directory, ...)` against the same `directory` (`App/Transcription/TranscriptionService.swift:677-683`). The generation guard prevents stale state assignment, not stale disk writes. The 500 ms grace is not a correctness barrier.
3. Likely: app foreground can auto-start recording without a tap, and can do so while prior post-stop transcription/cleanup is still running. On every active scene, `handleSceneActive()` calls `attemptAutoStart` (`App/JotApp.swift:72-75`, `App/JotApp.swift:124-145`). The guard only checks setup/model/mic permission and `!recordingService.isRecording` (`App/JotApp.swift:135-145`), so after `stop()` has drained audio but before transcription/cleanup finishes, foregrounding can start a new recording under an in-flight transcript tail.

## New issues introduced by iter 2
1. Confirmed: main-app cleanup ignores user/persisted cleanup configuration. It always passes `CleanupSettings.defaultInstructions` (`App/ContentView.swift:236-241`), while the persisted contract is `CleanupSettings.load()` with `enabled` and `instructions` (`Shared/CleanupSettings.swift:17-21`) and the intent pipeline still respects `cleanup.enabled` (`App/Intents/DictationPipeline.swift:187-195`). This creates behavior divergence across main-app and intent dictation.
2. Confirmed: `SettingsView` no longer exposes cleanup settings and now only has `TranscriptionService` in environment (`App/Settings/SettingsView.swift:3-7`). Combined with the main-app unconditional default cleanup above, users have no visible way to disable or tune the cleanup path that now gates saving.
3. Possible: strict-concurrency/build status is still unknown. `xcodegen generate` succeeded, but `xcodebuild` did not reach Swift compilation in this sandbox: package resolution failed on SwiftPM manifest sandboxing (`sandbox-exec: sandbox_apply: Operation not permitted`) and CoreSimulatorService was unavailable. No confirmed new MainActor/Sendable violation was found by inspection, but this is not a compiler-verified statement.

## Verdict
Another iter is needed before sim re-verification/device test. C2 and C3 look mechanically correct, and S2 is at least more observable, but C1 and S3 still leave production-grade failure modes: cleanup can indefinitely block persistence, and purge/reload can still race an uncancellable FluidAudio download at the filesystem level. The new foreground auto-start behavior also needs an explicit product decision or stronger gating before this is safe to hand to device testing.
