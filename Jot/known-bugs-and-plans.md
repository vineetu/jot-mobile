# Jot — Known Bugs & Planned Work

> Companion to **[features.md](features.md)**. This page holds the **bug tracker** and the **plan / roadmap index** — the implementation-level detail (symptoms, hypotheses, plan-doc links, sizes) that is intentionally kept OUT of the feature inventory, which stays user-facing only. `§N.M` references link back into `features.md`.

---

## Sizing convention

Entries below carry a **size** tag using a T-shirt scale:

- **XS** — under 2 hours, single-file trivial change.
- **S** — 2-6 hours, single feature surface, 1-3 files.
- **M** — 1-2 days, multi-file but contained.
- **L** — 3-5 days, multi-component, real design decisions.
- **XL** — over a week, architectural change or new subsystem.

Sizes appear on the Known Bugs below and on aspirational entries that haven't shipped. Already-shipped features don't carry sizes.

### Keyboard "Added to Vocabulary" confirmation uses error styling and appears late
**Size: S.** **Plan: [docs/plans/bug-keyboard-overlay-surfaces.md](../docs/plans/bug-keyboard-overlay-surfaces.md).** Mockup: atlas screen `kb-status-banner` (flagged). Relates to [§5.6](features.md#5-6-actions-popover) / [§5.10](features.md#5-10-status-banner).

**Symptom:** Selecting a word in the keyboard and tapping **Add to Vocabulary** (••• Actions popover) shows the confirmation "Added '<word>' to your dictionary" in the **red error banner with an ✕ icon** — a success message rendered as an error. The banner floats over the top of the recents pane. Two defects: (1) wrong tint/icon (should be a success/green style, not the error red); (2) it does **not** appear instantly — it only surfaces after a follow-up action (e.g. pressing Enter / the next keystroke), so the confirmation is decoupled from the action that triggered it.

**When it happens:** Every time a word is added to Vocabulary from the keyboard popover.

**Current hypothesis:** the banner reuses a single error-styled status-banner component for all messages (no success variant), and the banner text is written to the App-Group `lastDictationStatusMessage` projection but the keyboard only re-reads/renders it on the next input event rather than reacting immediately to the add — so it shows late. Needs confirmation in `JotKeyboardViewController.refreshStatusBanner` / `setStatusBanner` and the add-to-vocab path.

### Keyboard Actions popover displaces the Recents list instead of floating over it
**Size: S.** **Plan: [docs/plans/bug-keyboard-overlay-surfaces.md](../docs/plans/bug-keyboard-overlay-surfaces.md).** Mockup: atlas screen `kb-actions-popover` (flagged). Relates to [§5.6](features.md#5-6-actions-popover).

**Symptom:** Opening the ••• Actions popover renders the menu **inside the keyboard pane** (top-left) and pushes the [Recents strip](features.md#5-2-recents-strip-idle-state) down out of view, instead of floating over the recents as a context menu. The user loses the recents context while the popover is open, and the popover's lower rows (Undo/Redo) can be clipped by the control row.

**When it happens:** Every time the Actions popover is opened.

**Current hypothesis:** the popover is laid out as a sibling in the keyboard's vertical stack (taking layout space and pushing siblings down) rather than as an overlay (`.overlay`/`ZStack` anchored above the ••• key). Fixing it to a floating overlay would preserve the recents underneath.

### Keyboard-initiated dictation opens Jot but does not start recording
**Size: S.** **Plan: [docs/plans/bug-cold-start-dictation-race.md](../docs/plans/bug-cold-start-dictation-race.md).** Hypothesis recorded here may not match code reality — see plan for diagnostic-first approach.

**Symptom:** User taps the Dictate pill in the Jot keyboard from a host app. Jot is brought to the foreground (recording hero may briefly appear) but recording never actually starts — the timer stays at 0:00 and no audio is captured. The user has to back out and tap the home-screen Dictate FAB to actually record.

**When it happens:** Cold-start path only (no [warm-hold](features.md#13-2-warm-hold) window active). Observed especially after the speech model has been unloaded (e.g. post-memory-warning or first launch after device reboot). User manually triggering dictation in the same session briefly flashes a "Parakeet loading…" indicator before recording begins, which suggests the keyboard-initiated path may be racing the speech-model ANE-load.

**Current hypothesis:** the `jot://dictate` URL-bounce reaches the main app before the speech model has finished loading into ANE. The recording-start handler proceeds and starts the audio engine, but the streaming session's `loadModels(from:)` fails or hangs, leaving the recording in a half-started state where the engine isn't capturing. Needs trace logs to confirm.

### Auto-paste silently fails in some host apps (notably Slack)
**Size: M. Status: ✅ FIXED via PROXY RE-SYNC — build 108 (confirmed on-device).** **Plan: [docs/plans/bug-slack-silent-paste.md](../docs/plans/bug-slack-silent-paste.md).** The REAL root cause (Update #9) is host-dependent: a cold `insertText` no-ops into a custom/web compose field (Claude/Slack) that re-mounted during the record→transcribe gap, leaving the proxy's input connection stale — works in native fields (Messages). Builds 104/105 (notification-race fix, Updates #6–#7) were an earlier, different chapter that 106 regressed. **See Update #9 below for the definitive cause + fix.** **NB history: build 103 made it worse; 104/105 fixed the notification race; 106 ("land once") regressed host-paste; 108 re-sync fixes the actual stale-proxy cause.**

**Symptom:** Stop a warm-hold dictation while focused on a host app's text field (observed in Slack). The keyboard's `pasteSuccess` event fires in the diagnostics log with the full character count, but no text appears in the host field. The dictated transcript is still stored in Jot's library — only the auto-paste step silently no-ops.

**When it happens:** Warm-hold path (Jot never foregrounded). Most reliably observed with Slack as the host; possibly other apps with custom JS-backed compose fields or draft-autosave systems. Intermittent rather than every time.

**Why `pasteSuccess` is misleading:** `UITextDocumentProxy.insertText(_:)` is a void call — it returns nothing whether the host accepted the insert or not. The "pasteSuccess" log line confirms only that the keyboard *called* the function, not that any text actually landed in the host's text field. The host can silently reject the insert if the proxy is mid-disconnect, if the keyboard isn't the active input view at the moment of call, or if the host's text-input delegate rejects programmatic inserts.

**Current hypothesis:** the host's textDocumentProxy is disconnected (returns `nil` for `documentContextBeforeInput`) at the moment of insert, so iOS no-ops the call silently. Slack's compose field has known unusual focus + draft-autosave behavior that may transiently break the proxy connection. As of build 4 the `pasteSuccess` log now also captures `proxyHadContextBefore` + a before/after context-length delta so the next failure produces actionable signal: if `contextGrew` equals the inserted character count, the host accepted the text; if it's `0` or `-1`, the insert was silently dropped. Once a failure is captured with the new instrumentation, the fix (likely gating insert on `proxyHadContextBefore == true` and posting an error banner otherwise) becomes a one-liner.

**Update 2026-06-05 #6 — ✅ FIXED & SHIPPED (build 103). #5's "silent insert-drop" was ALSO wrong — `contextGrew` is a false negative on long pastes.** A clean 4-stop log settled it: `documentContextBeforeInput` returns a bounded, line-reset window, so a long/multi-line insert makes the reported pre-caret context SHRINK even though the text landed (owner confirmed the 309-char `contextGrew=-256` stop "this time it pasted"). The genuine failure had **no `pasteSuccess` at all** — the insert never ran. **Real root cause:** the auto-paste flush (`flushPendingAutoPasteIfPossible`, does NOT gate on phase) is driven ONLY by `pipelinePhaseChanged`. In the publish sequence (`DictationPipeline` fresh branch) the ONLY phase post that fires AFTER the payload is written is the terminal `publishPipelinePhase(.idle)` — which also calls `stopHeartbeat()`, so it's the LAST notification for the session. Darwin posts of the same name coalesce; when that single `.idle` post is coalesced/dropped (it trails a same-named burst + an `await transitionPostPublish`), the flush never re-runs, the heartbeat is already off → the published payload sits unread forever. One missed notification = permanent silent loss → intermittent. Proof: failing stop `558E7EEF` logged `publishCompleted` then total silence (heartbeat stopped = `.idle` ran, but its notif didn't wake the flush); identical-shape `E07DED06` got the `.idle` flush 4ms later → pasted. **Fix (`JotKeyboardViewController`): also drive `flushPendingAutoPasteIfPossible()` from a `transcriptReady` observer.** `transcriptReady` is posted the instant the payload is written (no await), in the live notification window — a reliable "transcription done → paste" trigger; `.idle` stays as backstop; double-flush is idempotent (`markConsumed` + `clearPendingPasteSession`). The old "don't observe transcriptReady → stale recents" objection applies only to the history mirror, not paste (payload already written). Build 103 → TestFlight (UPLOAD SUCCEEDED). **All prior theories here (suspension #3, publishCompleted-stall #4, insert-drop #5) were wrong — the bug was a coalescing race on the single post-payload notification.**

**Update 2026-06-05 #7 — ✅ DEFINITIVELY FIXED (build 104), confirmed on-device. #6's mechanism was right but its `transcriptReady` fix (build 103) MADE IT WORSE.** A build-104 on-device experiment (fire the insert on every trigger, tag each with which trigger + whether it landed, and keep the transcript until one lands) gave the clean answer: a single stop logged `trigger=transcriptReady` → `proxyHadContextBefore=false, landed=false` (proxy NOT connected at the early moment), then ~4ms later `trigger=pipelinePhase` (settled `.idle`) → `proxyHadContextBefore=true, contextGrew=50==chars, landed=true` LANDED. So **the paste lands on `pipelinePhase` (`.idle`, the settled moment) — exactly what worked before build 103.** Build 103's regression was ADDING the `transcriptReady` trigger: it fired the insert into the not-yet-connected proxy at the stop instant AND the flush unconditionally consumed the payload (`markConsumed`+`clearPendingPasteSession`), so the good `.idle` attempt found nothing → deterministic fail every time. **Build 104 fix (shipped):** don't consume the transcript unless the insert verifiably landed (`landed = proxyHadContextBefore || contextGrew > 0`); the early failed attempt now keeps the payload so the settled `.idle` trigger lands it. Plus a re-entrancy guard, a `textDidChange` backstop, and (experiment scaffolding) a 6-attempt cap + per-trigger diagnostic logging. **CLEANUP DONE (build 105):** removed the `transcriptReady` trigger entirely; the keyboard now fires the paste ONLY on `pipelinePhase` (`.idle`). Kept the one robustness lesson — the don't-consume-unless-`landed` guard (`landed = proxyHadContextBefore || contextGrew > 0`): a disconnected-proxy insert no longer burns the transcript, it stays pending. Single trigger = zero double-paste risk (no cap needed). Ripped out the experiment scaffolding (6-attempt cap, attempt counter, `isAutoPasteFlushing` re-entrancy flag, `textDidChange` retry trigger, and the `trigger`/`landed`/`attempt` diagnostic metadata). The not-landed case now logs under `pasteSkipProxyDisconnected`. `contextGrew` is no longer trusted as the success oracle (false-negative on long/multi-line pastes — `documentContextBeforeInput` returns a bounded, line-reset window); `proxyHadContextBefore` is primary.

**Update 2026-06-08 #9 — ✅ FIXED via PROXY RE-SYNC (build 108), confirmed on-device. The 104/105 fix (notification race) was a real but DIFFERENT bug; build 106 ("land once, reliably") regressed host-paste and exposed the ACTUAL root cause.** Owner symptom that cracked it: auto-paste fails in **Claude's app + many apps but WORKS in Messages**, and at stop the field is provably LIVE (cursor blinking, transcript already shown in keyboard recents). So it is NOT a dropped notification and NOT "stale because the user navigated away" — it is a **host-dependent insert no-op**. Mechanism (corroborated by FluidAudio source + Apple docs + our own coalescing code): the transcript arrives ~hundreds of ms after Stop (record → transcribe → cross-process publish), NOT inside a UI event; in that gap a **custom/web compose field (Claude/Slack = React-Native/WKWebView) re-mounts its text view**, so `textDocumentProxy` points at a stale input connection — the pointer still looks valid and the caret still blinks, but a **cold `insertText` silently no-ops**. A native `UITextView` (Messages) keeps the connection, so it always lands there. There is **no iOS API to keep another app's field/proxy alive** (a keyboard extension is handed a proxy, never the field) — confirmed via research; the only lever is to RE-SYNC before writing. **Build 108 fix (`flushPendingAutoPasteIfPossible`):** issue `adjustTextPosition(byCharacterOffset: 0)` (caret unchanged) to force the host to re-establish the input connection, then **yield ONE run-loop tick** (`DispatchQueue.main.asyncAfter +0.012`) BEFORE the insert — because iOS COALESCES `adjustTextPosition` into the current UI cycle (our `moveUpStep`/`moveDownStep` already proved a synchronous nudge gets swallowed), so a sync nudge-then-insert still hits the stale link. A new `isAutoPasteInsertInFlight` guard covers the ~12ms window so a second phase-change flush can't stack a duplicate → **single insert, no retry band-aid** (owner explicitly rejected retry as a "double-charge"). Logs now carry `resynced=true`. The build-106 landed-detection (`proxyHadContextBefore || proxyHasContextAfter`, keep-pending if not landed) is preserved as the safety net.

### Minimize sometimes leaves keyboard at full height with collapsed content inside
**Size: diagnosis-first.** **Plan: [docs/plans/bug-keyboard-minimize-stuck-height.md](../docs/plans/bug-keyboard-minimize-stuck-height.md).** Ten plausible causes enumerated; fix size XS→M depending on which one confirms via the existing `[KB-COLLAPSE-DEBUG]` instrumentation.

**Symptom:** Tapping the Minimize button in the [Jot Keyboard](features.md#5-8-minimize--expand) sometimes swaps the SwiftUI render to the collapsed-bar view (small Stop button centered) but leaves the keyboard's outer frame at full expanded height (~450 pt). User sees the small content floating in a tall empty envelope. Intermittent — not every Minimize tap is affected.

**Workaround that works:** globe-switch to any other keyboard, then globe-switch back to Jot. On re-presentation, the keyboard renders at the correct collapsed 58 pt height.

**This is the opposite failure mode from [§5.10](features.md#5-10-status-banner):** there the height grows but content doesn't; here the content shrinks but height doesn't.
