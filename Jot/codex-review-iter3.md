## Critical issues remaining

1. Confirmed: F2 can still strand large model-cache directories after an interrupted purge. `purgeAndReload()` creates a fresh UUID-suffixed purging directory for the current purge (`App/Transcription/TranscriptionService.swift:246`, `App/Transcription/TranscriptionService.swift:747`), but the only deletion path targets that one directory in `Task.detached` (`App/Transcription/TranscriptionService.swift:756`). A crash, jetsam, or suspension during that detached delete leaves the old `.purging-*` directory behind; the next purge creates another UUID dir rather than reclaiming existing ones. The settings UI documents the model as about 1.25 GB (`App/Settings/SettingsView.swift:34`), so repeated failed deletes are not benign.

2. Possible: F1 does not prove that cleanup generation stops at 8s; it proves that Jot stops awaiting it and sends cancellation. The timeout path cancels `cleanupTask` in `defer` (`App/ContentView.swift:322`) and the bridge cancellation handler (`App/ContentView.swift:30`), while `CleanupService.clean` checks cancellation before `respond` and catches `CancellationError` (`App/Cleanup/CleanupService.swift:149`, `App/Cleanup/CleanupService.swift:156`). There is no hard stop around `LanguageModelSession.respond(to:)` itself (`App/Cleanup/CleanupService.swift:151`). If that API does not promptly honor task cancellation, the unstructured `Task { @MainActor in ... }` can continue holding the transcript/session after raw fallback has already been persisted.

## New issues introduced

1. Likely: the purge fix trades the original live-directory delete race for an orphan-cache risk. The UUID suffix avoids same-name collisions in practice (`App/Transcription/TranscriptionService.swift:751`), but there is no startup or pre-purge cleanup for prior `*.purging-*` siblings. Search performed: `rg "purging-|deletePurgedModelDirectory|purgingModelDirectory|purgeAndReload" App Shared` found only the current move/delete helpers, not a sweeper.

2. Possible: AppIntent phase can briefly report `.idle` between `DictationControllerImpl.stopAndTranscribe()` returning and `DictationPipeline.completeEndOfRecording()` calling `beginPostProcessing()`. The controller sets `currentPhase = .idle` in a `defer` (`App/Intents/DictateIntent.swift:413`, `App/Intents/DictateIntent.swift:417`), and the pipeline only re-enters post-processing after the caller invokes it (`App/Intents/DictationPipeline.swift:117`). The new `isPipelineInFlight` guard prevents an actual overlapping recording (`App/Recording/RecordingService.swift:95`), but a rapid toggle in that gap may surface `alreadyRunning` instead of no-oping cleanly.

## Verdict

Another iter is needed before I would call this ready for sim re-verification + device test. F3 and F4 are code-level passes. F1 is acceptable as a user-facing fallback but still needs device evidence that FoundationModels cancellation actually stops generation promptly. F2 is not closed: atomic move protects the live path, but interrupted detached deletion can leak one full model cache per failed purge. Add a pre-purge/startup sweeper for stale `.purging-*` siblings, and preferably wrap deletion in a background task or otherwise make the cleanup resumable.
