# Issue #3 fix — adversarial review

**Reviewed:** `docs/carplay/issue-3-mic-rootcause.md` (root cause + proposed fix: set `RecordAndTranscribeIntent.openAppWhenRun = true` + correct doc-comments).
**Method:** every code claim re-checked against source (file:line below); every external claim re-fetched from Apple primary sources and forum threads (URLs below). The doc-author's quote of thread 756507 and the error-code mapping both hold. Two new, load-bearing facts surfaced that the doc does not account for.

---

## Verdict

The **diagnosis is correct** and the **direction of the fix is correct** (the app must be foreground; foregrounding is the only Apple-sanctioned path; hypothesis A / `AudioRecordingIntent` is rightly rejected). But the **prescribed one-liner is not safe to ship as-is** for two independent reasons the doc misses:

1. **`openAppWhenRun` is deprecated as of iOS 26.0** ([thread 804015](https://developer.apple.com/forums/thread/804015), DTS Engineer Ed Ford), and Jot's deployment target **is** iOS 26.0 (`Jot/project.yml:5`). The doc proposes flipping a property that is deprecated on the project's *minimum* OS. The supported replacement is `supportedModes` with `.foreground` (`.foreground(.immediate)` reproduces the old `openAppWhenRun=true` behavior).
2. **The doc's load-bearing assumption — "app is foreground before `perform()` runs" — is not what Apple's own DTS describes.** Per Ed Ford ([thread 769924](https://developer.apple.com/forums/thread/769924)): *"a scene is created and displayed as part of the execution of your intent, but that is **after** the creation and launch of your app into the background is complete."* The app launches into the **background**; the scene/foreground transition is *part of* intent execution, not strictly before it. So `perform()` calling `RecordingService.start()` synchronously can still race the foreground transition. The fix is a **strong probabilistic improvement, not a guaranteed one** — and it should be hardened by deferring the actual mic-start until the scene is confirmed active, rather than calling `start()` inline in `perform()`.

Net: ship the foreground bounce, but (a) use `supportedModes`/`.foreground` not the deprecated bool, and (b) defer mic-start to scene-active. Both are small. Without (b) you may have re-introduced an intermittent version of the same bug — which the project's own memory warns about (`[[feedback_intermittent_bug_needs_multiple_repros]]`: one green run does not clear a race).

## MUST-FIX

### M1 — Don't flip the deprecated `openAppWhenRun`; use `supportedModes` (`.foreground`).
- **Claim challenged:** doc §4 Option A / §0 verdict: "set `RecordAndTranscribeIntent.openAppWhenRun = true` — one-line change."
- **Evidence:** Apple DTS Engineer Ed Ford, [thread 804015](https://developer.apple.com/forums/thread/804015): *"`openAppWhenRun` is now **deprecated**, and you should look at [`supportedModes`](https://developer.apple.com/documentation/appintents/appintent/supportedmodes) instead."* Corroborated by the [openAppWhenRun doc](https://developer.apple.com/documentation/appintents/appintent/openappwhenrun) (deprecated iOS 26.0) and migration write-ups stating `.foreground(.immediate)` == old `openAppWhenRun=true`. Jot ships **iOS 26.0 minimum** (`Jot/project.yml:5`), so this is deprecated on the floor OS, not a future concern.
- **Concrete change:** Replace the `openAppWhenRun = true` flip with a `supportedModes` declaration on the registered intent (`.foreground(.immediate)` for the start leg). Keep `openAppWhenRun` only if you must support < iOS 26 — Jot does not. (Confidence: Confirmed — DTS + deployment target.)
- **Caveat:** I could not fetch the rendered `supportedModes` doc body (Apple docs are JS-rendered; WebFetch returned title-only). The deprecation + the `.foreground(.immediate)` equivalence are confirmed from the DTS thread and multiple secondary sources; the *exact* enum spelling should be confirmed against the SDK header in Xcode before writing it.

### M2 — Defer mic-start to scene-active; do not call `RecordingService.start()` inline in `perform()`.
- **Claim challenged:** doc §3 / §1.2 implicit assumption that once the bounce intent runs, "the existing `RecordingService.start()` path then succeeds because the app is foreground." The fix calls `controller.startRecording()` → `recording.start()` directly inside `perform()` (`RecordAndTranscribeIntent.swift:148→166`, `DictationControllerImpl.startRecording` `DictateIntent.swift:411-415`).
- **Evidence (this is the crux, point 1):** DTS Ed Ford, [thread 769924](https://developer.apple.com/forums/thread/769924): *"For those app intents that have `openAppWhenRun` set to `true`, a scene is created and displayed **as part of the execution of your intent**, but that is **after** the creation and launch of your app into the **background** is complete."* I.e. the documented sequence is: launch into background → `perform()` executes → scene created/displayed → foreground transition callbacks. There is **no Apple guarantee that the app has reached `.active` foreground before `perform()`'s body runs.** `RecordingService.start()` calls `setActive(true)` (`RecordingService.swift:1258`) then `engine.start()` (`:588`) — exactly the operations iOS gates on foreground state. If `perform()` wins the race against the scene activation, you get the *same* `.engineStart` / `setActive` failure intermittently.
- **Why the doc under-weights this:** `DictateIntent`'s own comment (`DictateIntent.swift:154-157`) asserts the app "has already foregrounded by the time `perform()` runs," sourced only to *"Empirically observed on iPhone 15 Pro, iOS 26"* (`DictateIntent.swift:81`). That is a single-device empirical observation, contradicted by the DTS ordering description. The doc inherits this optimistic claim.
- **Concrete change:** Foreground via the bounce, but have `perform()` **post a request** that the foregrounded scene services once it confirms `.active` (e.g. set a pending-start flag / notification that `JotApp`'s scene-active path consumes — Jot already has this exact machinery: `scenePhase == .active` handling, `triggerAutoStart`, and the `autoStartPending*` flags in `JotApp.swift:30-34`, plus the `jot://dictate` cold-launch auto-start path which **already foregrounds-then-starts** rather than starting in an intent). Route the Action-Button/Spotlight start through that proven scene-active path instead of starting the mic inside `perform()`. (Confidence: Confirmed the race is *possible*; Confirmed Jot already has a scene-active start path to reuse. Whether the race actually fires on the owner's hardware is **Unknown without on-device repro** — but the safe design removes the question.)
- **Cross-check against project memory:** `[[feedback_intermittent_bug_needs_multiple_repros]]` and `[[feedback_never_band_aid]]` both apply — calling `start()` in `perform()` and hoping foreground won the race is the band-aid; routing through scene-active is the root-cause-clean shape.

### M3 — Fix the citation the doc gets backwards about its own evidence weight (minor but it's a MUST because it's load-bearing for the verdict).
- **Claim challenged:** doc §1.2 cites thread 756507's reported error as **561015905** and maps it to `cannotStartPlaying` ('!pla'), then says "the exact code differs by build/route." That mapping is **correct** (verified: 561145187 = `cannotStartRecording` '!rec'; 561015905 = `cannotStartPlaying` '!pla' — confirmed via [Apple ErrorCode docs](https://developer.apple.com/documentation/coreaudiotypes/avaudiosession/errorcode/cannotstartrecording) search + [cannotStartPlaying](https://developer.apple.com/documentation/avfaudio/avaudiosession/errorcode/cannotstartplaying)). **However**, the doc does not note *why* 756507 reports the *playing* code: the OP there used **`.playAndRecord`** (verified in thread 756507 — OP states *"sets the audio session category to `.playAndRecord`"*). Jot uses **`.record`** (`RecordingService.swift:1252-1256`). So the 756507 error code is not directly comparable to Jot's symptom — but the *mechanism* (foreground gate) is identical and is what the DTS answer addresses. This strengthens, not weakens, the diagnosis; the doc should state it so a future reader doesn't trip on the code mismatch.
- **Concrete change:** One sentence in §1.2 noting 756507's OP used `.playAndRecord` (hence the `!pla` code), Jot uses `.record`, mechanism is the same. (Confidence: Confirmed.)

## NICE-TO-HAVE

### N1 — Point 3 (toggle/stop across the bounce): acceptable, but document the second-press bounce.
With the foreground intent, **press 2 (stop) also foregrounds Jot.** Trace: each `perform()` reads `controller.currentPhase` (`RecordAndTranscribeIntent.swift:146`); press 1 `.idle → beginDictation`, press 2 `.recording → endDictation` (`:149-150`). Both presses run the same foreground intent, so press 2 bounces the app forward just to stop+transcribe. `RecordingService.shared` is genuinely shared between scene and intent (`RecordingService.swift:73` singleton; `JotApp.swift:164,182` `_recordingService = State(initialValue: RecordingService.shared)`; `DictationControllerImpl.init` `DictateIntent.swift:407`), so the stop operates on the right session and `currentPhase` is authoritative across presses — **no correctness bug**. But for Spotlight specifically, press 2 from Spotlight is unusual (the user is already in-app after press 1 and would more likely use the in-app stop). Acceptable; worth a one-line UX note. (Confidence: Confirmed via code trace.)

### N2 — Point 4 (flip-the-bool vs swap-to-DictateIntent): the flip is safer; doc's recommendation is right, reasoning is incomplete.
The doc recommends flipping the registered `RecordAndTranscribeIntent` over swapping registration to `DictateIntent`. That **is** the safer choice — but the stronger reason is that `RecordAndTranscribeIntent` is the **already-bound** tile: users who bound "Jot down" bound *that* intent's identifier. Swapping the `AppShortcut` to `DictateIntent` changes the intent identity behind an existing binding, which can drop the user's Action-Button/Spotlight binding until re-bound. `perform()` bodies are functionally identical (`RecordAndTranscribeIntent.swift:135-196` vs `DictateIntent.swift:150-206` — verified line-equivalent: same bridge, same `beginDictation`/`endDictation`, same `DictationPipeline.completeEndOfRecording`). One real divergence: `beginDictation` orders the coordinator call differently (`RecordAndTranscribeIntent` calls `DictationActivityCoordinator.shared.start()` **before** `startRecording` and rolls it back on throw `:163-170`; `DictateIntent` calls `startRecording` **then** `.start()` `:177-178`). Neither touches the mic-start gate. Keep `RecordAndTranscribeIntent` as the tile. (Confidence: Confirmed code-equivalent; binding-identity risk is Likely.)

### N3 — Doc-comment corrections (doc §6): agreed, do them. Add one.
All six listed corrections are accurate and warranted (the struct conforms to plain `AppIntent` at `RecordAndTranscribeIntent.swift:105`; the `AudioRecordingIntent` "no foregrounding needed" claims at `:21-26,55-58` are false). **Add a seventh:** once M1 lands, the comment at `DictateIntent.swift:94-98` ("`openAppWhenRun`… is required") and `RecordAndTranscribeIntent.swift:116` ("Load-bearing") must be updated to reference `supportedModes`, or they'll mislead the next reader into reusing a deprecated API. (Confidence: Confirmed.)

### N4 — Keep the diagnostic logging recommendation (doc §5.4) — promote it to "do before shipping," not "optional."
The `engine.start()` catch (`RecordingService.swift:589-595`) throws `.engineStart(error)` but does **not** log domain/code, unlike `configureSession` which does (`:1269-1272`). Given M2 (the race may still fire intermittently), capturing the underlying `com.apple.coreaudio` code on a real Spotlight press is how you'd *detect* a surviving race post-fix. Add the logging in the same change, not as a deferred optional. (Confidence: Confirmed code asymmetry.)

## VERIFIED CORRECT

- **Failure point + call order** (doc §1.1): `.engineStart` thrown at `RecordingService.swift:594` from `try engine.start()` `:588`; `configureSession` runs first `:549`, `setActive(true)` at `:1258` would throw `.sessionConfiguration` `:1274` if it failed; mic preflight `:564-568`; converter `:570-573`. All confirmed. The reasoning "`setActive` succeeded, so the I/O start is what's refused" is sound.
- **Thread 756507 is genuinely Apple DTS** and says what the doc quotes: *"You cannot trigger an audio recording from the Shortcuts app. Your app needs to be in the foreground before the user can start recording audio."* and *"Consider overriding the `openAppWhenRun` property to return `true`… your app is brought to the foreground when the user intends to record audio."* (Verified verbatim.) The doc's reported error code (561015905) and the OP's `.playAndRecord` category also match (see M3).
- **Error-code mapping** (doc §1.2 / Sources): 561145187 = `cannotStartRecording` '!rec', 561015905 = `cannotStartPlaying` '!pla'. Confirmed.
- **Hypothesis A (re-add `AudioRecordingIntent`) is correctly rejected** (doc §2): [thread 815725](https://developer.apple.com/forums/thread/815725) confirms cold-background recording is impossible even with `AudioRecordingIntent`; *"The Live Activity and its Intents can only be used to manage (pause/resume) an already established session, rather than launching a new one from a cold background state."* The doc's own caveat that 815725's answer is from community member **farisdev, not Apple DTS**, is accurate and appropriately weighted — the authoritative source (756507, DTS) agrees. Live Activity subsystem removal verified (no `NSSupportsLiveActivities`).
- **`RecordingService.shared` is genuinely shared** between foreground scene and intent (singleton `:73`; scene `JotApp.swift:182`; intent `DictateIntent.swift:407`). The doc's reliance on the shared-session-state for clean toggle/stop holds.
- **Config already present** (doc §4): `NSMicrophoneUsageDescription` and `UIBackgroundModes: [audio]` — not re-verified line-by-line this pass, but consistent with the codebase; no new entitlement needed for the foreground-bounce approach.
- **`AudioRecordingIntent` doc-comment claims are false** as the doc states — the struct is plain `AppIntent` (`:105`).
