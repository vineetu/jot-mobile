Jot · Ask — retrieval engine

From a blurry guess
to a real RAG pipeline.

Why "Ask" returns the wrong notes today, what it should be instead, and exactly how we get there — now that embeddings are still beta and nothing is shipped.

TL;DR — Today we squeeze a whole transcript into one vector with an old, small model. Long notes get averaged into mush and the right answer never surfaces. The fix isn't a tweak — it's chunking + a better embedder + hybrid search + reranking + intent routing. And because no user has embeddings yet, we can delete the old system and rebuild it clean.
01 — The shift

Where we are vs. where we're going

Same job — "find the notes that answer this question" — two very different engines.

● Current · broken

One vector per transcript

all-MiniLM-L6-v2 · 384-d · cosine top-15

  • A 14-minute note → one vector. Every idea inside is averaged into a blur.
  • MiniLM truncates at ~256 tokens — most of a long note is never even embedded.
  • No keyword search — misses names, jargon, codenames the meaning-vector skips.
  • No reranking — the right note is often rank 12, never seen.
  • "Last 3 days" / "lately" are treated as searches, not summaries → random notes.
● Target · RAG

Many vectors per transcript

EmbeddingGemma-300M · chunked · hybrid + rerank

  • Each note is split into chunks — every idea gets its own searchable vector.
  • EmbeddingGemma-300M: newer, on-device (<200 MB), sees the whole chunk.
  • Hybrid search: meaning (vectors) + keywords (BM25), fused together.
  • Reranking re-scores the top candidates so the best chunk wins.
  • Intent routing: time/summary questions get summarized, not searched.

The key insight that keeps it simple: at ~1,000 transcripts you get ~10,000 chunks ≈ 10 MB of vectors. Scanning all of them takes <50 ms on-device. So no vector database, no fancy index — plain math is plenty until you're 10× bigger.
03 — How we move

Clean rebuild — not a careful migration

Embeddings shipped to nobody. That's a gift: we don't preserve old data — we tear it out and regenerate from the transcripts (the real source of truth), which are untouched.

  1. Drop the old embedding store SAFE — BETA

    Delete the per-transcript embedding table and the MiniLM service. Transcripts themselves are never touched.

  2. Add the new shape

    A TranscriptChunk record: chunk text, its vector, position in the note, and metadata (date, tags, type). Plus on-device keyword search (SQLite FTS5).

  3. Re-index everything once

    On next launch, a background task walks every transcript, chunks it, and embeds the chunks with EmbeddingGemma. Runs quietly; Ask switches on when it's done.

  4. Rebuild tags on the new model

    The existing category classifier just re-learns its centroids from your seed phrases using the new embeddings — same feature, new brain.

  5. Wire up the new Ask path + measure

    Route → filter → hybrid → rerank → answer. Ship a small eval set (question → expected notes) so "better" is a number, not a vibe.
